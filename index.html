Aegis Sanctions Screening

Aegis is Booking.com's centralized Sanctions Screening platform. It protects the business from regulatory threats by screening individuals, businesses, and financial assets against global sanctions lists.

ⓘ Who is this integration guide for?

Managers — understand compliance flows and regulatory requirements.

Engineers — implement the screening integration into your service.

Choosing an Integration Path

Select the integration type that matches your screening needs. Each path is optimized for different volumes, frequencies, and latency requirements.

Scheduled

Batch Screening

For ongoing relationships that need daily screening — partners, hotels, vendors.

Tech: CSV + REST API / Java Client Response: Asynchronous (daily/nightly)
Ad-hoc

Ad-hoc Screening

For unscheduled bulk tasks — large datasets at irregular intervals, payout reports.

Tech: Kafka (Event-driven) Response: Asynchronous (eventual consistency)
Realtime

Realtime Screening

For immediate screening — booker reservation requests, instant payout initiation.

Tech: REST API (Synchronous) Response: Immediate + async updates

Batch Screening

For partners, vendors, and affiliates that require ongoing scheduled screening.

Batch screening uses a synchronization model. You don't ask Aegis to "screen this partner" — you tell Aegis "here is the current state of this partner," and Aegis handles daily screening against the latest sanctions lists.

Key concept: You own the data, Aegis owns the screening schedule.

Common Use Cases

  • Daily Partner Screening — Ensure all active partners (ABU, TBU) are screened every 24 hours.
  • Payout Screening — Screen all bank details linked to partners before payouts are processed.

Phase 2: Add / Update API

Once bootstrapped, use the API to ADD new entities or UPDATE existing ones. This endpoint is asynchronous.

ADD

Sync newly onboarded data with Aegis so it is screened at the pre-defined frequency.

UPDATE

Send when any screening-relevant data changes (e.g., partner updates their name).

Important: You must send the entire record (and any linked records) as part of the update call, not just the fields that changed.

Ad-hoc Screening

For bulk analysis, irregular reporting, and large datasets (100+ records).

Ad-hoc screening is event-driven. You produce a message to a Kafka topic, and Aegis consumes, screens, and produces a result message back to an outbound topic.

Match Statuses

Every screening result returns one of four statuses:

SCREENED

No hits found. Safe to proceed with onboarding or payout.

POTENTIAL_MATCH

Possible hit. Do not payout or onboard. Manual review triggered internally.

TRUE_MATCH

Confirmed sanction hit. Must block immediately.

FALSE_POSITIVE

Flagged but cleared by analyst. Safe to proceed.

Status Handling Example 
// Pseudocode: handling match status

switch (result.overall_status) {
  case "SCREENED":
    proceed();
    break;

  case "POTENTIAL_MATCH":
    holdTransaction();
    // Wait for analyst review
    break;

  case "TRUE_MATCH":
    blockImmediately();
    escalate();
    break;

  case "FALSE_POSITIVE":
    proceed();
    break;
}

Realtime Screening

For immediate screening at the point of action — onboarding, reservations, payments.

Unlike Batch (scheduled) or Ad-hoc (event-driven), the Realtime API provides an initial synchronous response, optionally followed by asynchronous updates based on analyst review.

Common Use Cases

  • Reservation Requests — Check a guest or booker at the moment of reservation.
  • Instant Data Updates — Verify a change in user details immediately.
  • Payor Screening — Check the payer name at the time of payment.

Error Codes

Errors returned in the screening response indicate issues with your request or downstream systems.

INVALID_DATA Fix & Retry

Data is malformed, incomplete, or contains invalid values — missing required fields, incorrect data types, malformed UUIDs.

DUPLICATE_DATA No Action Needed

The entity (source, externalId, requestId) already exists. The original record will be processed.

VALIDATION_ERROR Fix & Retry

Data fails business validation — invalid field combinations, business logic constraints, field length violations.

VENDOR_ERROR Under Review

Error from the screening vendor (e.g., name doesn't meet minimum screening requirements). Contact RegTech for guidance.

UNKNOWN_ERROR Retry Later

Unexpected system error during processing. May be handled internally — retry after some time.

Error Response Shape 
{
  "request_id": "req-uuid-555",
  "external_id": "TBU-IND-005",
  "match_summary": null,
  "errors": [
    {
      "code": "INVALID_DATA",
      "message": "Missing required field: full_name"

  ]
}

Support & Compliance

Non-Functional Requirements

Retries

Your integration must implement exponential backoff retries for 4xx/5xx API errors.

Logging

Do not log PII in your application logs when debugging Aegis errors.

Test Data

Do not use real PII in DQS environments. Use the Skeleton Test Plan provided by RegTech.

Retry Implementation 
// Exponential backoff example

const MAX_RETRIES = 5;

async function screenWithRetry(payload) {
  for (let i = 0; i < MAX_RETRIES; i++) {
    try {
      return await fetch(URL, {
        method: "POST",
        headers: {
          "X-Booking-Authentication": token
        },
        body: JSON.stringify(payload)
      });
    } catch (err) {
      if (i === MAX_RETRIES - 1) throw err;
      await sleep(Math.pow(2, i) * 1000);
    }
  }
}
